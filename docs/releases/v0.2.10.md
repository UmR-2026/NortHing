# Northhing v0.2.10 — Frontend Onboarding

> **Release date**: 2026-06-26
> **Cycle**: v0.1.0 frontend onboarding (spec v1.2)
> **Base**: v0.1.0 (2026-06-24 MVP — P0+P1 debt cleared)
> **Commits**: 20

This release finishes the desktop frontend. The product core has
been ready since v0.1.0; what was missing was the surfaces to
configure it, the flows to start using it, and the error channels
to make the daily-use pain go away. v0.2.10 lands all of those.

---

## What's new

### Settings panel — configure everything the core already supports

The Settings view (gear icon, bottom-left of the sidebar) now
exposes the four configuration surfaces that the v0.1.0 backend
was waiting for:

- **AI 服务** (Provider) — add, edit, delete, test provider
  configurations. Legacy cleanup banner warns if a provider
  with no enabled models is detected. New providers can be
  added inline without leaving the view.
- **技能** (Skills) — list of installed skills, with global
  enable / disable and per-workspace override (three-state:
  default / always on / always off).
- **工具集 (MCP)** — full MCP server management across all
  three transports (stdio / http / sse), with environment
  variables and a one-click test action.
- **工作文件夹** (Workspace) — workspace list with a native
  folder picker, IDENTITY.md entry point, and a confirmation
  dialog for the destructive "remove" action.
- **通用** (General) — global app preferences.

### Welcome flow — first-run experience

On first launch (or after deleting `app.json`), the app now
shows a 3-step Welcome view instead of the empty main view:

1. Pick a workspace folder.
2. Create or import an IDENTITY.md (5-round LLM Q&A, optional).
3. Confirm and start using the app.

Subsequent launches go straight to the main view.

### Chat enhancements

- **Tool calls** in the chat pane are now collapsible (was
  flat).
- **Inline error channel** — when an LLM call fails or a
  session becomes unusable (e.g. its provider was removed),
  the error is shown directly in the chat pane with a ×
  button to dismiss. The user does not have to switch sessions
  to see what went wrong.
- **Banner error channel** — transient / summary errors
  (e.g. "Settings save failed") appear as a top-of-window
  banner with a "详情" button that copies the same error to
  the inline channel. Auto-clears after 5 seconds.
- **Session header** — each session has a header row with
  settings, export, and stop actions.
- **Workspace switcher** in the sidebar (pill control) for
  fast switching between active workspaces.

### Relay security

- Room IDs are now 16 bytes of cryptographic entropy (was
  6 bytes, predictable).
- Stale-room cleanup is race-free (retain + 2-pass strategy,
  2 unit tests).
- `pair` and `command` API endpoints now require an API key
  (set `RELAY_API_KEY` on the relay server).

## Bug fixes

### 9 Slint setter sites (same root cause)

Slint 1.16 silently drops property setters called from a
non-event-loop thread. Across this cycle, 9 call sites in
`app_state/mod.rs` were reached from background threads and
were silently dropping their updates. All 9 are now wrapped
in `slint::invoke_from_event_loop` which dispatches the
closure onto the UI thread.

The 9 sites and the user-visible symptom if they were not
fixed:

- Welcome route on first launch — first-launch users would
  see the main view instead of the Welcome view.
- Sidebar "设置" click — clicking the gear in the sidebar did
  nothing (no Rust handler was registered for the
  `open-settings` callback).
- `model-status` (Provider panel) — stayed at the
  "Not configured" placeholder after configuring a model.
- `mcp-status` (MCP panel) — stayed at the placeholder after
  configuring an MCP server.
- P0-A startup session — first-launch auto-created session
  existed in `core` but was not visible in the sidebar
  until the next manual refresh.
- Session error 5s auto-clear — never cleared, stayed on
  screen until the next user action.
- Input error 5s auto-clear — same.
- Banner 5s auto-clear — same.
- Banner detail 5s auto-clear — same.
- After AI reply, the message list did not refresh; the user
  had to switch sessions to see the new messages.

### Q6 / Q7 integrity checks (Phase 5 wire-up)

When a provider is deleted, any session that was using it as
its default model is flagged and the user is warned via the
banner + inline error channel. Same for workspaces: removing
a workspace marks any session bound to it as broken and
surfaces the issue in the UI. Previously these deletions
succeeded silently and the affected sessions would fail on
the next LLM call with a confusing error.

### Sidebar open-settings

The `open-settings` callback is now wired through the Slint
side directly (the Settings view's × button was already
pure Slint; the open path was previously asymmetric and went
through a Rust handler). One FFI boundary removed; behavior
unchanged.

## Upgrade notes

- **First-time users** will see the new Welcome flow.
  Existing users with an `app.json` will not see the Welcome
  flow and will go straight to the main view.
- **No migration** is required for the Settings data. The
  `app.json` schema is unchanged; the v0.1.0 backend and
  the v0.2.10 desktop read it the same way.
- **Relay operators** who upgrade to v0.2.10 will need to
  set `RELAY_API_KEY` if they want `pair` / `command`
  endpoints to require authentication. Without the env var,
  the endpoints reject all requests (safe default).
- **Slint padding warnings** (`padding only has effect on
  layout elements`) appear in the build output for
  `IdentityCreatorView.slint`, `WelcomeView.slint`, and
  `SidebarView.slint`. These are cosmetic and do not affect
  runtime behavior. Plan to address in a follow-up cycle.
