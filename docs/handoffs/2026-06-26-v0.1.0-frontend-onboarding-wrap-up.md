## 1. Why this cycle

The v0.1.0 MVP shipped 2026-06-24 (CHANGELOG `[0.1.0]` entry) with the
product core stable but the desktop app still a command-line shell.
For the A-phase goal ("切到 NortHing daily use" — switch from
WorkBuddy + Trae to NortHing as the daily tool), three pain points
needed frontend work:

- Subagent invocation: backend 100% ready, no UI entry point
- Subagent LLM switching: backend 100% ready, no UI control
- Long-horizon memory: backend 100% ready, no UI surfacing

Plus the `f4` "设置" (Settings) view was a true blocker — the product
core ships with 4 settings surface (Provider / Skills / MCP /
Workspace) and the desktop app could not configure any of them.

This cycle delivered the Settings panel, the dual-channel error
design, the welcome / identity / workspace flows, and the Q6/Q7
integrity checks that the v0.1.0 backend was waiting for.

## 2. Spec & process

Design philosophy locked in v1.2 of the spec:

- **Code-related surfaces as inspectable black boxes** — Provider /
  Skills / MCP lists are visible; subagent / memory / deep-review /
  long-horizon internals stay black-box (visible enough to inspect,
  not required to understand).
- **Rust-first → full-stack → AI assistant** (d2 + d3 + d5
  differentiation).
- **A short-term / B long-term** — the daily-use pain is A; B items
  stay in spec §12 for later cycles.

Spec went through three self-review rounds:

1. **v1.0 → v1.1** — caught 4 spec/plan errors (goal_mode numbering,
   round_preempt naming, Deep Review owner crate, Computer Use 4-way
   split, cursor_format) and 1 planning error (the original
   roadmap over-counted by 6 tasks).
2. **v1.1 → v1.2** — added §5.6 (data contract for `AppSettings`),
   §5.7 (AppSettings ↔ ConfigManager boundary), §5.8 (Slint ↔ Rust
   bridge, 5 patterns), §6 (Implementation Patterns, 26
   self-decided conventions C-i through C-xxvi).
3. **35 grill-me decisions + 13 self-review decisions** locked into
   §4 of the spec.

Spec v1.2 was the source of truth for everything that landed.

## 5. Bug fixes (9 sites, same root cause)

All 9 are the same bug: a Slint property setter is called from a
non-event-loop thread, where Slint 1.16 silently drops the update.
The fix pattern in every case is `slint::invoke_from_event_loop`
which dispatches the closure onto the UI thread. The set of
callers also had to be cross-checked for `Send` requirements
(`ModelRc` is `!Send`, so any setter that takes a model must be
constructed on the UI thread, not moved across boundaries).

### Welcome route (`748f628`)

- `create_ui` background thread called
  `ui.set_current_route("welcome")` for first-launch.
- Wrapped the setter in `invoke_from_event_loop`.
- Symptom if not fixed: first-launch users would see the main
  view instead of the Welcome view.

### Sidebar open-settings wire (`97890b2`)

- `main.slint` declared `callback open-settings()` and the
  sidebar chained `open-settings => { root.open-settings(); }`,
  but no Rust `ui.on_open_settings` handler was ever registered.
- Fixed by adding the 5-line handler that flips
  `current-route` to "settings".
- This was the original Phase 2 wiring bug; W-01 manual test
  caught it.
- (The handler was later removed in `5b7deeb` when the cycle
  collapsed both `open-settings` and `close-settings` to pure
  Slint. The behavior fix is preserved in `5b7deeb` via the
  Slint-side change.)

### 3 latent bugs from `0f05cbc` review (`bff005a`)

| Site | Visible symptom if not fixed |
|---|---|
| Phase C.3 model-status refresh | model-status stays at "Not configured" placeholder after configuring a model |
| Phase G.2 mcp-status refresh | mcp-status stays at placeholder after configuring an MCP server |
| P0-A startup session OK branch | sidebar session list not refreshed after first-launch auto-create session; session exists in `core` but UI shows empty list until next manual refresh |

Out of scope (deliberately not fixed in `bff005a`): the 3 P0-A
`Err` branches (`set_session_error` from background thread).
Less impactful (only fires on error paths that are already
rare) and not flagged by the reviewer. Left for a future pass.

### 5 sites from `a4f3ac3` cleanup review (`29a72eb`)

| Site | Visible symptom if not fixed |
|---|---|
| `schedule_error_clear` Session auto-clear | Session error banner never auto-clears after 5s |
| `schedule_error_clear` Input auto-clear | Input error banner never auto-clears after 5s |
| `schedule_error_clear` Banner auto-clear | Top-of-window banner never auto-clears after 5s |
| `schedule_error_clear` Banner-detail auto-clear | Banner detail text never auto-clears after 5s |
| Response-complete `set_messages` | After AI reply completes, message list not refreshed; user has to switch sessions to see the new messages |

The 4 `schedule_error_clear` sites got a simple
`invoke_from_event_loop` wrapper. The response-complete refresh
was restructured: the outer `std::thread::spawn` is no longer
needed because the dispatched closure spins up the runtime
itself and runs everything on the UI thread (ModelRc must be
constructed on the UI thread; this is the only legal shape).

### Net effect

9 setter call sites fixed, 0 new ones introduced. The
`slint::invoke_from_event_loop` pattern is now established
and documented in two places (the P0-A comment in `bff005a` and
the response-complete comment in `29a72eb`); future code in
this codebase should use it for any Slint setter called from
a background context.
