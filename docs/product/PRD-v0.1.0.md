## 1. 产品定位

### 1.1 一句话描述

**northhing** 是一个**纯桌面 GUI 应用**，作为个人 AI 助手通过自然语言对话帮助用户完成编码、文件管理、知识整理和自动化任务。所有交互通过图形界面完成，CLI 仅作为内部机制供 agent 和子 agent 调用，不向人类用户暴露。

### 1.2 核心架构原则

| 原则 | 说明 |
|------|------|
| **人类用户 → GUI 唯一入口** | 所有用户操作通过 Slint 桌面界面完成，无 CLI 入口 |
| **Agent → 内部函数调用** | Agent 和子 agent 通过 Rust API 调用，非 shell 命令 |
| **本地优先** | 数据存储在本地 SQLite，LLM 调用走云端 API |
| **单用户架构** | 无需多用户、权限、审计等团队功能 |
| **进程内隔离** | 子 agent 通过线程/任务隔离，非独立进程 |

### 1.3 核心差异（vs. v3 团队协作版）

| 维度 | v3 (BitFun) | northhing (个人版) |
|------|-------------|-------------------|
| **目标用户** | 开发团队、企业 | 个人开发者、技术爱好者 |
| **核心场景** | 团队协作、PR 审查、合规治理 | 个人编码、知识管理、自动化 |
| **架构重心** | 多用户、权限、审计、治理 | 单用户、本地优先、快速响应 |
| **UI 形态** | Tauri + React 桌面壳 | **Slint + Material 桌面壳** |
| **功能范围** | 全功能（mini-apps、PPT、podcast） | 核心功能（聊天、文件、工具、技能） |
| **部署模式** | 团队服务器 + 个人客户端 | **纯本地，LLM 走云端** |
| **数据存储** | 团队共享、云端同步 | **本地 SQLite** |
| **扩展机制** | 插件市场、团队配置 | **技能目录、MCP 工具** |
| **CLI 暴露** | 完整 CLI 供用户使用 | **CLI 隐藏，仅内部使用** |
| **Agent 调用** | 可 shell 调用 | **内部函数调用** |

### 1.3 用户画像

| 画像 | 描述 | 核心需求 | 使用频率 |
|------|------|---------|---------|
| **独立开发者** | 全栈工程师，管理多个个人项目 | 代码生成、文件操作、项目理解 | 每日 |
| **技术研究者** | 学习新技术、整理笔记、实验代码 | 知识整理、文档生成、代码解释 | 每周 |
| **自动化用户** | 用脚本和工具简化重复工作 | 工具调用、批量操作、定时任务 | 按需 |
| **多项目管理者** | 同时维护多个开源/商业项目 | 会话隔离、项目切换、上下文保持 | 每日 |

## 2. 产品目标

### 2.1 必达目标 (MVP v0.1.0)

1. **本地对话**: 用户打开应用，选择模型，开始自然语言对话
2. **文件感知**: 应用能读取、编辑、搜索用户项目文件
3. **工具调用**: 支持文件操作、shell 命令、代码搜索等内置工具
4. **多会话**: 支持多个独立会话，每个会话有自己的上下文
5. **多模型**: 支持任意 OpenAI-compatible API（OpenAI, Anthropic, Ollama, vLLM 等）
6. **技能扩展**: 从 `~/.config/northhing/skills/` 加载 Markdown 技能文件，按需注入提示

### 2.2 期望目标 (v0.2.0)

1. **远程工作区**: 通过 SSH 连接远程服务器进行开发
2. **MCP 工具**: 支持 Model Context Protocol 工具调用
3. **子代理**: 会话中可派生专项子代理处理特定任务
4. **知识库**: 本地文档索引和 RAG 查询
5. **定时任务**: 支持 cron 式定时执行自动化工作流

### 2.3 愿景目标 (v1.0.0)

1. **插件生态**: 第三方技能和能力扩展市场
2. **团队协作**: 可选的团队共享会话和知识库（保持本地优先）
3. **多模态**: 图像理解、语音输入、代码可视化
4. **自适应**: 基于用户习惯自动优化提示和工具选择

## 3. 核心功能需求

### 3.1 会话系统 (Session)

#### 3.1.1 会话生命周期

```
创建会话 → 选择模型/技能 → 发送消息 → 接收回复 → 工具调用 → 结果反馈 → 继续对话 → 关闭/保存
```

#### 3.1.2 需求列表

| ID | 需求 | 优先级 | 验收标准 |
|----|------|--------|---------|
| SE-01 | 创建新会话 | P0 | 点击"+"按钮，输入会话名称，选择模型，进入对话 |
| SE-02 | 会话列表 | P0 | 左侧边栏显示所有会话，按时间倒序，支持搜索 |
| SE-03 | 会话切换 | P0 | 点击会话项，立即切换上下文，不丢失历史 |
| SE-04 | 会话持久化 | P0 | 关闭应用后重启，所有会话和消息恢复 |
| SE-05 | 删除会话 | P0 | 支持删除单个或批量删除，有确认提示 |
| SE-06 | 会话重命名 | P1 | 右键或点击标题编辑会话名称 |
| SE-07 | 会话导出 | P1 | 导出为 Markdown 或 JSON 格式 |
| SE-08 | 子代理 | P2 | 在会话中派生子代理处理专项任务，结果合并回父会话 |

#### 3.1.3 数据结构

```rust
pub struct Session {
    pub id: SessionId,           // UUID
    pub name: String,            // 用户可编辑
    pub parent: Option<SessionId>, // 子代理指向父会话
    pub model_provider: String,   // "openai", "anthropic", "ollama"
    pub model_id: String,        // "gpt-4o", "claude-3-5-sonnet"
    pub skill_set: Vec<String>,  // 已加载的技能 ID 列表
    pub messages: Vec<Message>,  // 消息历史
    pub created_at: DateTime,
    pub updated_at: DateTime,
    pub metadata: SessionMetadata, // token 使用、工具调用次数等
}

pub struct Message {
    pub id: MessageId,
    pub role: Role,              // User, Assistant, System, Tool
    pub content: String,
    pub tool_calls: Vec<ToolCall>, // 助手请求的工具调用
    pub tool_results: Vec<ToolResult>, // 工具执行结果
    pub timestamp: DateTime,
    pub tokens: Option<TokenUsage>,
}
```

### 3.4 技能系统 (Skills)

#### 3.4.1 技能定义格式

```markdown

id: rust-debugging
name: Rust 调试助手
version: 1.0.0
description: 帮助诊断和修复 Rust 编译错误和运行时问题
triggers: ["rust", "编译错误", "borrow checker", "lifetime"]
---

# Rust 调试助手

当你遇到 Rust 编译错误时，按以下步骤诊断：

1. 读取错误信息，定位具体文件和行号
2. 检查变量所有权和生命周期
3. 建议使用 `clone()`, `Rc`, `Arc`, 或重构代码
4. 提供修复后的代码示例

## 常用诊断命令

```bash
cargo check          # 快速检查
cargo clippy         # 详细建议
cargo test --lib     # 运行单元测试
```
```

**v0.1.0 决策**: 采用文件系统技能存储，而非硬编码或数据库存储。理由：
1. 用户可版本控制（Git）技能文件
2. 社区可分享技能（复制 Markdown 文件）
3. 编辑简单（任何文本编辑器）
4. 无需数据库迁移

## 5. 架构决策

### 5.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | Slint + Material | 声明式 UI，跨平台，原生性能 |
| 后端 | Rust | 类型安全，性能优秀，生态丰富 |
| 存储 | SQLite | 轻量，本地优先，无需服务器 |
| 配置 | TOML | 人类可读，支持注释 |
| 协议 | HTTP | LLM 通信，简单可靠 |
| 扩展 | MCP | 工具标准化协议 |

### 5.2 模块划分

```
northhing/
├── crates/
│   ├── ui/slint-shell/          # 桌面 UI 壳 (A1)
│   ├── cli-internal/            # 内部 CLI (A3)
│   ├── core/                    # 核心聚合 (A2)
│   ├── contracts/
│   │   ├── core-types/          # 基础类型
│   │   ├── events/              # 事件系统
│   │   └── runtime-ports/       # 运行时接口
│   ├── execution/
│   │   ├── agent-runtime/       # 代理运行时
│   │   ├── tool-contracts/      # 工具契约
│   │   ├── tool-execution/      # 工具执行
│   │   └── harness/             # 工作流 harness
│   ├── adapters/
│   │   ├── ai-adapters/         # LLM 适配器
│   │   ├── transport/           # 传输层
│   │   └── api-layer/           # API 层
│   ├── services/
│   │   ├── services-core/       # 核心服务
│   │   ├── services-integrations/ # 集成服务
│   │   └── terminal/            # 终端服务
│   └── assembly/
│       └── core/                # 核心组装
└── apps/
    └── cli/                     # 公共 CLI (A3 重写为内部 CLI)
```

### 5.3 数据流

```
用户输入 → UI 层 → 会话管理 → 代理运行时 → LLM 适配器 → 模型 API
                                    ↓
                              工具调用 → 工具执行 → 文件系统/Shell/网络
                                    ↓
                              结果返回 → 消息渲染 → UI 展示
```

### 5.4 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 本地优先 | SQLite + 本地文件 | 隐私、速度、离线可用 |
| 声明式 UI | Slint | 性能、类型安全、设计师友好 |
| 技能即 Markdown | 文件系统存储 | 版本控制、社区分享、简单编辑 |
| 工具标准化 | MCP 协议 | 生态兼容、供应商中立 |
| 异步运行时 | Tokio | 并发、性能、生态 |
| 错误处理 | anyhow + thiserror | 简洁、可组合 |

## 7. 验收标准

### 7.1 功能验收

| 场景 | 步骤 | 期望结果 |
|------|------|---------|
| 首次启动 | 打开应用 | 显示欢迎页，提示选择项目目录 |
| 开始对话 | 选择模型 → 输入消息 | 收到助手回复，流式显示 |
| 文件操作 | 请求"读取 main.rs" | 显示文件内容，有语法高亮 |
| 工具调用 | 请求"运行 cargo test" | 显示命令确认，执行后显示结果 |
| 多会话 | 创建 3 个会话，分别对话 | 每个会话独立上下文，切换不丢失 |
| 持久化 | 关闭应用，重启 | 所有会话和消息恢复 |
| 技能加载 | 在 skills/ 目录添加技能文件 | 对话中触发关键词，技能内容注入 |
| 模型切换 | 在设置中添加 Ollama | 新会话可选择本地模型 |

### 7.2 性能验收

| 测试 | 条件 | 通过标准 |
|------|------|---------|
| 冷启动 | 空项目 | < 3 秒 |
| 大文件读取 | 1MB 日志文件 | < 500ms |
| 搜索 | 1000 文件项目 | < 1 秒 |
| 内存泄漏 | 运行 1 小时 | 内存增长 < 50MB |
| 并发会话 | 10 个活跃会话 | 切换流畅，无卡顿 |

### 7.3 安全验收

| 测试 | 步骤 | 通过标准 |
|------|------|---------|
| API Key 存储 | 配置 OpenAI Key | 存储加密，明文不可见 |
| 危险命令 | 请求"rm -rf /" | 弹窗确认，默认拒绝 |
| 网络访问 | 请求访问未知域名 | 弹窗确认，可拒绝 |
| 跨项目访问 | 会话 A 访问会话 B 的文件 | 路径隔离，拒绝访问 |

## 8. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| Slint 生态不成熟 | 高 | 中 | 准备 fallback 到 egui，保持 UI 层薄 |
| LLM API 变更 | 中 | 高 | 适配器模式，快速响应变化 |
| Windows 兼容性问题 | 中 | 中 | CI 覆盖 Windows，优先测试 |
| 性能瓶颈（大项目） | 中 | 中 | 增量索引，异步加载，分页 |
| 用户隐私担忧 | 高 | 低 | 本地优先，数据不离开本机，透明说明 |
| 技能质量不一 | 低 | 中 | 社区审核，评分系统，官方精选 |

## 9. 附录

### 9.1 术语表

| 术语 | 定义 |
|------|------|
| **Session** | 一次独立的对话上下文，包含消息历史和配置 |
| **Skill** | Markdown 格式的领域知识文件，按需注入系统提示 |
| **Tool** | 可执行的功能单元，如文件操作、命令执行 |
| **MCP** | Model Context Protocol，工具标准化协议 |
| **Provider** | LLM 服务提供商，如 OpenAI, Anthropic |
| **Workspace** | 用户打开的项目目录，northhing 在此上下文中工作 |
| **Subagent** | 从主会话派生的专项代理，处理特定任务后返回结果 |
