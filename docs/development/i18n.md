# I18n Development Guide

Use this guide for day-to-day i18n changes. Architecture and ownership rules are
in `docs/architecture/i18n.md`.

## Quick Rules

- Edit language metadata only in `src/shared/i18n/contract/locales.json`.
- Run `pnpm run i18n:generate` after changing the contract or shared terms.
- Keep stable cross-surface labels in
  `src/shared/i18n/resources/shared/<locale>/terms.json`.
- Keep workflow, page, toast, validation, and feature copy in the owning
  surface namespace.
- Do not import Web UI locale catalogs into mobile-web, installer, backend, or
  static pages.
- Store resources as readable UTF-8. Use escapes only when JSON, regex, or code
  syntax requires them.
- Do not add static `defaultValue` fallback copy in components, including
  composed static `defaultValue` expressions. Add or repair locale keys instead.
- Use `defaultValue` only for real runtime data fallbacks, such as
  server-provided names or descriptions.
- Use surface i18n formatting helpers for user-visible dates, times, numbers,
  and currency; avoid direct `Intl.*` or `toLocale*` calls in product code.
- Persist and send canonical locale ids: `zh-CN`, `zh-TW`, or `en-US`.

## Commands

Run the smallest set that matches the files you touched. CI protects broad
builds and suites.

```bash
pnpm run i18n:generate
pnpm run i18n:contract:test
pnpm run i18n:audit
```

| Change | Local check |
|---|---|
| Locale resource-only JSON or Fluent changes | `pnpm run i18n:audit` |
| Locale contract or shared terms | `pnpm run i18n:generate && pnpm run i18n:contract:test && pnpm run i18n:audit` |
| Web UI i18n runtime or namespace loading | `pnpm run i18n:contract:test && pnpm run type-check:web && pnpm --dir src/web-ui run test:run src/infrastructure/i18n/core/I18nService.test.ts` |
| Mobile Web i18n runtime | `pnpm --dir src/mobile-web run type-check` |
| Installer frontend i18n runtime | `pnpm --dir northhing-Installer run type-check` |
| Backend i18n runtime | `cargo test -p northhing-core i18n -- --nocapture` |

Do not add process-only execution plans to version control. Keep durable rules
in this file and `docs/architecture/i18n.md`; keep temporary rollout notes out
of git.

## Where Strings Belong

Use shared terms for stable labels whose meaning must stay aligned everywhere:

- `product.*`
- `features.*`
- `modes.*`
- `agents.*`
- `tools.*`
- `connectionMethods.*`
- `statuses.*`

Use surface common namespaces for short atoms that are generic inside that
surface. Use feature namespaces for workflow text, field labels, validation,
empty states, toasts, and page titles.

Shared terms are exposed per surface:

- Web UI: `t('shared:features.deepReview')`
- Mobile Web: `t('shared.features.deepReview')`
- Installer: `t('shared.features.deepReview')`
- Backend: `I18nService::translate*("shared.features.deepReview", ...)`
- Relay homepage: `{ "$shared": "features.remoteControl" }`

`shared` is not an implicit fallback namespace. If a local key may fall back to
a shared concept, keep the fallback explicit and local-first.

## Key Policy

JSON resources use dot-separated semantic keys with camelCase segments. Prefer
stable meaning over current English copy:

```text
sessions.startRemoteFlow
connectionMethods.northhingServer
tools.explore
```

Avoid positional or copy-shaped keys:

```text
button1
f1t
areYouSureYouWantToDelete
```

Allowed exceptions are i18next plural suffixes, documented runtime enum or
protocol mirrors, and existing backend Fluent kebab-case keys.

Before deleting a key, check static calls, namespace-relative hooks, metadata
fields such as `labelKey`, fallback arrays, string concatenation, template
literals, backend mappings, and static self-contained pages. Delete only
confirmed unused keys.

## Loading Rules

Unifying the language contract must not unify all translation bundles.

- Web UI eagerly loads only `WEB_UI_BOOTSTRAP_NAMESPACES`. Route and feature
  copy should load through `useI18n(namespace)` or `useTranslation(namespace)`.
- Mobile Web, Installer, Backend, and relay static pages own their own resources
  and must not import Web UI catalogs.
- Large or rarely used Web UI copy should live in lazy feature namespaces, not
  bootstrap namespaces.

## Audit Signals

`pnpm run i18n:audit -- --report-json <path>` separates:

- confirmed unused keys
- source-proven dynamic key candidates
- shared-term duplicate candidates
- broad same-text locale inventory
- l10n quality candidates
- literal fallback candidates
- direct locale-format candidates

The reports are review inputs. Only confirmed unused keys are deletion evidence.
Dynamic keys need source-proof allowlist entries. Shared-term duplicates should
move only when they represent the same stable concept. Same-text locale
inventory is non-blocking review data; do not treat a smaller actionable l10n
candidate count as proof that all localization work is complete.

L10n quality candidates are actionable same-text locale pairs with script or
terminology signals. Values that are intentionally written the same across
locales should either remain out of the signal set or be narrowly recorded in
`scripts/i18n-l10n-identical-allowlist.json`.

Keep these no-growth guards tight:

- `scripts/i18n-governance-baseline.json`
- `scripts/i18n-literal-fallback-baseline.json`
- `scripts/i18n-locale-format-baseline.json`

Lower baselines when debt is removed. Do not raise them unless the PR explicitly
accepts the risk.

## PR Expectations

Every i18n PR should state:

- changed surfaces
- whether the locale contract or generated files changed
- which focused i18n checks passed
- whether any surface intentionally does not support a locale
- any baseline change, with a short reason
