# northhing 可配置开发体验与工程治理架构设计

> 产品输入：[product-requirements.md](product-requirements.md)
> 范围：把产品需求转为架构边界、领域模型、配置优先级和模块职责。用户画像、体验路径和功能需求以产品需求文档为准— ## 1. 设计目标

| 目标 | 架构要求 |
|---|---|
| 快速路径轻— | 项目打开、任务执行和结果摘要不依赖完整证据包或图— |
| 用户语言稳定 | 内部策略画像不直接外露，统一投影为任务状态、弱提示、确认和设置 |
| 策略可解— | 每次提示、升级、阻断和覆盖都有来源、原因和适用范围 |
| 阶段收益可交— | 架构能力按用户收益切片落地，技术跑道、API 预设和测试桩不伪装成用户价— |
| 安全独立 | 执行安全与质量治理分层，安全边界优先于用户覆盖和质量模式 |
| 配置可组— | 用户、任务、工作区、路径、团队和组织策略按稳定优先级合成 |
| 证据可追— | PR、发布、事故和合规场景可以引用证据包、质量事件和交付物图— |
| 评估可回— | 策略、工具、模型、上下文和安全版本能进入评测与指标分— |
| 生态可扩展 | 先在 Rust 内核预留 hook、事件、打点和工具复写候选契约；外部插件生态后续通过适配器接— |

## 2. 复杂来源

| 来源 | 例子 | 架构要求 |
|---|---|---|
| 结构复杂 | 单体多项目、多语言、多服务、生成代码、跨仓库依赖 | 渐进项目画像和路径级规则 |
| 流程复杂 | issue、spec、PR、发布、事故分散在多个系统 | 交付物图谱后台关联，按场景投— |
| 验证复杂 | 私有依赖、不稳定 CI、本地环境缺— | 替代验证建议和不可验证解— |
| 风险复杂 | 权限、网络、凭据、迁移、发布、远程工作区 | 安全边界和守护策略分层处— |
| 团队复杂 | CODEOWNERS、组织策略、合规审— | 团队治理和风险接— |
| 信息复杂 | 旧文档、— 突规则、prompt 注入、未知配— | 来源、信任、新鲜度和— 突状— |

## 3. 顶层领域模型

| 领域 | 核心对象 | 作用 |
|---|---|---|
| 项目理解 | 工作区、仓库、执行环境、远程上下文、语言、框架、模块、规则来源、验证能— | 减少用户解释和错误路径选择 |
| 用户画像 | 用户类型、任务熟悉度、偏好、授权范围、受管状— | 选择默认入口、解释语言和提示密— |
| 任务与意— | 任务、阶段、目标路径、会话模式、用户覆— | 决定快速、PR、发布或应急路— |
| 阶段收益 | 用户可见收益、必要技术前置、延期边界、降级解释、质量一致性检— | 约束阶段交付必须产生可解释体验变— |
| 体验投影 | 内部策略画像、用户可见状态、提示层级、确认策略、设置项 | 降低用户学习成本和提示噪— |
| 执行安全 | 权限、执行位置、沙箱等级、网络域名、凭据访问、主动配置信任、应急放— | 防止 prompt、配置或工具越权 |
| 内核扩展— | 生命周期 hook、规范事件、打点、工具复写候选、插件效果候— | 为后续多插件生态适配预留稳定契约 |
| 变更信心 | 变更摘要、验证摘要、风险提示、未验证项、跳过检— | 给用户可理解的信心和下一步行— |
| 团队治理 | 路径规则、审查强度、强制检查、风险接受、审计策— | 统一团队和高可靠场景体验 |
| 生命周期上下— | issue、spec、计划、PR、发布、事故、学习资— | 支撑复杂项目追溯和复— |
| 评测与学— | 轨迹回放、评测任务、判定标准、反馈、指— | 证明策略改善体验和质— |

## 4. 逻辑架构

```text
用户界面
 快速工作台 / 变更摘要 / PR 就绪— / 发布审查 / 设置

体验投影— 任务状— / 弱提— / 可折叠摘— / 确认 / 受限原因 / 降噪规则

阶段收益编排
 用户收益 / 技术前— / 延期边界 / 降级解释 / 质量一致性检— 配置化策略面
 意图识别 / 策略画像 / 风险提示 / 审查画像 / 覆盖策略

安全边界
 执行位置 / 沙箱等级 / 权限 / 网络 / 凭据 / 主动配置信任 / 应急放— 内核扩展— 生命周期 hooks / 规范事件 / 打点 / 工具复写候— / 插件效果候— 未来插件适配主机（按需启用— 适配器注册表 / 项目执行— / 插件单元 / 工具门面 / 事件路由 / 候选效果输— 交付物与证据— 证据— / 交付物图— / 问题生命周期 / 风险接受

质量数据— 生命周期事件 / 执行轨迹 / 验证摘要 / 指标 / 本地审计

项目集成— Git / CI / Issue / 文档 / 发布 / 观测 / 知识— 智能体与工具运行— 会话 / 智能— / 终端 / 文件系统 / MCP-LSP / Hooks / 适配— ```

关键边界— - 用户界面展示用户可理解的任务状态、摘要、提示和确认— - 体验投影层把内部策略画像映射为用户语言，并负责提示合并、延后和降噪— - 阶段收益编排约束每次落地必须说明用户可见收益、后台前置、延期边界和质量一致性— - 配置化策略面决定内部体验强度、检查建议、证据展示层级和用户覆盖选项— - 安全边界负责权限、执行位置、沙箱等级、网络、凭据和高风险动作隔离；权限确认、快— 回滚隔离和运行时沙箱必须分开表达— - Kernel 扩展点先提供空实现和稳定契约；未来插件适配主机— Rust 内核管理生命周期，只返回建议、证据候选、工具参数补丁或工具结果候选。内核仍是状态、安全、工具执行和审计的唯一事实源— - 交付物、证据和质量数据面支撑解释、审查、发布和复盘— - 项目集成面适配外部系统，并把外部语义映射为内部稳定事件和契约— - 智能体运行时执行任务，并通过策略面和安全边界获取质量结论与授权状态— ## 5. 模块边界

## 6. 配置层级

| 层级 | 例子 | 作用 |
|---|---|---|
| 用户偏好 | 用户选择— 严、语言、默认授权范— | 个体体验默认— |
| Session/任务覆盖 | “这次快速放行网络”“本任务只读分析— | 临时治理强度 |
| 工作区配— | `.northhing/config.toml`— `.northhing/quality.yaml` | northhing 专属模式、检查、安全策— |
| 现有仓库规则 | AGENTS.md、CONTRIBUTING、CODEOWNERS、CI、`.github/instructions` | 项目知识和规则来— |
| 工具特定规则 | `.coderabbit.yaml`、`.gitlab/duo/*`、`.kiro/steering/*` | 外部工具和路径级经验 |
| 组织策略 | 受管配置、受保护分支、企业策— | 强制策略和不可绕过限— |

```text
组织拒绝 / 受管强制要求
 > 安全边界
 > 最近确认的路径或团队规— > 工作区配— > 允许范围内的会话/任务覆盖
 > 全局/用户默认— ```

## 7. 硬约— | 约束 | 设计要求 |
|---|---|
| 内部画像不直接外— | `fast/assist/review/guarded/regulated` 作为内部契约；用户侧展示任务状态、原因和下一— |
| 提示先弱后强 | 默认行内提示、可折叠摘要和任务结束汇总；弹窗只用于安全、不可逆、组织强制或关键流程 |
| 门禁按配置显— | 门禁— PR、团队、强策略或合规场景显— |
| 安全边界独立执行 | prompt 注入、凭据、网络、hook 等风险走独立判定 |
| 沙箱能力显式分层 | 本地、远程、ACP、MCP、插件、浏览器/桌面和云端任务都输出执行位置、沙箱等级和降级原因 |
| 阶段按收益切— | 同一模块可以跨阶段交付；每个阶段必须有用户可见收益、必要技术前置、延期边界和质量一致性检— |
| 技术契约后台化 | 证据包、质量数据面和图谱服务摘要、审查、发布和复盘 |
| 未知先进入建议— | 新项目初期先提示和建议，安全或团队策略再阻断 |
| 用户可临时放— | 应急放行有范围、期限、记录和撤销 |
| 组织策略可强— | 受管策略高于本地覆盖 |
| 模型输出作为候— | 确定性证据、用户决策和策略改变状— |
| 能力/效果模型统一 | tool、MCP、skills、插件、hook 和内置能力必须映射为能力声明、目标对象、数据类别、信任来源和副作用候— |
| 未声明能力受— | 新增扩展未声明能力、声明不完整或运行时行为超出声明时，只能进入受限模式或安全确认，不能按低风险静默执行 |
| 策略不写死工具名 | 策略引擎以能力、效果、数据、来源、执行域和配置上下文判定；工具名只用于展示、审计、兼容和调试 |
| 外部适配不写权威状— | 外部插件和适配器只能返回候选效果；通过、失败、阻断、审计和权限状态由内核写入 |
| 工具复写显式授权 | 用户确认复写哪个工具和能力范围；复写表按项目执行域生效，不能静默覆盖全局工具 |

## 8. 架构风险

| 风险 | 治理策略 |
|---|---|
| 产品默认过重 | P0 以快速路径和低摩擦指标验— |
| 技术计划脱离用户收— | 阶段评审必须检查用户可见收益、延期边界和质量一致性，后台技术只能作为明确前置项进入 |
| 内部术语外露 | 通过体验投影层统一用户语言，设置和日志才展示调试级策略信息 |
| 安全提示太频— | 沙箱、白名单、范围和域名策略降噪 |
| 应急放行被滥用 | 范围、期限、组织禁用、隔离建议和审计 |
| 项目规则被注入污— | 规则来源、信任、新鲜度和恶意指令检— |
| 路径规则— 突 | 最近规则、— 突展示和人工处理 |
| 扩展点时序漂— | 事件 id、sequence、deadline、epoch 和幂等键约束；过期响应不应用 |
| 能力声明被绕— | 适配器注册、工具复写、MCP 描述和插件效果都必须经过能力/效果校验；未知能力默认受— |
| 工具复写静默越权 | 内置工具复写必须显式展示、按项目生效，并重新经过安全边界 |
| 图谱和证据过早显— | 只在解释、PR、发布、事故时显性化 |
| 平均体验掩盖局部问— | 指标按用户画像、任务风险、内部策略画像、用户可见投影、入口平台和受管状态切— |
